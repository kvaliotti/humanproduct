# Structured Problem-Solving Backbone

This is the shared reasoning backbone for **every** skill in the PLG Growth plugin. Each skill applies these six principles to its own domain (acquisition, activation, retention, monetisation, satisfaction, revenue, readiness, loops, PLS, experimentation, data, transformation). Wherever a skill says "apply the backbone," it means walk these six principles against that skill's issue tree, metrics, and outputs.

## The Six Principles

1. **Issue Trees (MECE decomposition)** — Decompose the problem into mutually exclusive, collectively exhaustive branches. The decomposition must be complete (nothing important missing) and non-overlapping (each branch is distinct). Each skill decomposes *its own* problem space: acquisition into channels, monetisation into the pricing triangle, retention into its four components, and so on.

2. **Hypothesis Trees** — Before gathering data, form testable hypotheses at each branch about where the opportunity or the blocker lies. Hypothesize first, then go looking for the evidence that confirms or kills each hypothesis. This keeps analysis directed rather than boiling the ocean.

3. **Driver Disaggregation** — Break each metric into its drivers, separating:
   - **Direct / mathematical drivers** — the arithmetic components (x, +, −, ÷) that *define* the metric (e.g., ARPA = units × price; k-factor = invites × conversion).
   - **Indirect / behavioral drivers** — the correlational and behavioral factors that *influence* those components (e.g., motivation, ability, trigger; friction; perceived value).
   Disaggregation tells you which lever actually moves the number.

4. **80/20 Prioritization** — Focus on the 2–3 branches (or 1–2 for loop/model selection) with the highest sensitivity: the ones where a realistic change produces the largest outcome. Ignore low-leverage branches for now. Ruthless prioritization beats broad coverage.

5. **So-What Synthesis (Minto Pyramid)** — Communicate conclusion-first: lead with the recommendation / verdict / opportunity, then the supporting arguments, then the data. Never make the reader assemble the conclusion themselves.

6. **Hypothesis-Driven Work Plans** — Every recommended action ties back to a specific hypothesis from the issue tree. No orphan tasks: each work item exists to test or exploit a named hypothesis, with a clear "we'll know we're right if…" signal.

## How Skills Specialize This Backbone

The skeleton is constant; each skill reskins the domain-specific tail of principles 1–3:
- **Issue tree** is drawn from that skill's own framework (e.g., AARMS funnel, pricing triangle, four-component retention, TASE for data, the four transformation dimensions).
- **Driver disaggregation** uses that domain's math (k-factor and cycle time for loops; take rate × GMV or seats × price for monetisation; activation rate = signup → aha for activation).
- **Prioritization, synthesis, and work-plan** discipline (principles 4–6) are identical everywhere.

When in doubt, run the six principles in order. They are the connective tissue that makes the plugin's outputs consistent and decision-ready.
