# Lens: Behavioural Loop

**Distilled from:** *Atomic Habits* (Clear), *Tiny Habits* (Fogg), *Hooked* (Eyal). These three overlap ~70%, so they are merged here into **one** loop model with the redundancy removed. Where they used different words for the same thing — cue / prompt / trigger; response / action / ability; reward / celebration / variable reward; investment / tracking — this lens uses one vocabulary and scores each thing once.

## What this lens sees

A durable behaviour is a **loop**, not an event:

> **Trigger → Action → Reward → Investment**, repeating, getting easier and more valuable each cycle.

The loop only runs when, at the moment of the trigger, **B = MAP**: Motivation, Ability, and a Prompt converge above the action threshold. Diagnose failures in that order — **prompt first, then ability, then motivation** — because teams reflexively blame motivation when the real problem is a missing prompt or too-hard action. Most broken loops are broken on *ability* or *prompt*, and those are the cheapest to fix.

Score only what's relevant to this product's natural frequency. A loop that should fire daily (a habit tracker) is judged differently from one that fires quarterly (tax software) — for the latter, most habit machinery is **N/A**, and forcing it in is the bloat we're trying to prevent.

## 1. Target behaviour & loop existence  *(behaviour)*

- Is there ONE specific, observable target behaviour for this experience — not "engage more" but "user + situation + action + object + completion"? Two observers should agree it happened.
- Is it small enough to happen in a single session, and frequent enough (given the real user need) to plausibly repeat?
- Is there an actual loop, or just a one-shot funnel? Can you name the trigger, the action, the reward, and the investment — or do some links not exist?
- Is this the *golden* behaviour — high user value × high business value × high ease × existing motivation — or the first behaviour someone thought of? Was a cheaper/easier behaviour that drives the same outcome overlooked?
- Is there a "minimum viable" version of the behaviour that still counts as success on a low-energy day?

## 2. Trigger — does anything reliably start the loop?  *(engagement, adoption)*

- **External trigger:** is there a concrete prompt (notification, email, in-app cue, placement) that tells the user what to do next, points at exactly one action, and lands them where they can do it immediately? Or does the product assume the user will remember to come back on their own?
- **Internal trigger:** is the experience anchored to a recurring user emotion/need/itch (boredom, uncertainty, deadline pressure, curiosity, fear of missing something)? Can the user eventually recall the product *without* an external nudge — "this is what I use when I feel/need X"?
- Is the trigger close to the action in **time and context**, and tied to real user value rather than the company's desire for a re-open? Is it riding an existing routine/anchor ("after I ___, I ___") rather than asking the user to invent a new one?
- Are triggers overused to mask weak value (notification spam that trains users to ignore them)? Can the user tune or control them?
- Is the product deliberately moving users from paid/earned triggers → owned triggers → internal triggers over their lifecycle?

## 3. Action — is it easy enough to actually happen?  *(behaviour, adoption)*

Ability compensates for motivation: making the action easier is almost always cheaper and more reliable than trying to pump up motivation. Check what makes this action hard along each axis and whether the product has removed it:

- **Time, physical effort, mental effort, money/perceived risk** — is each minimized at the moment of action?
- Is the first meaningful action doable in seconds, before motivation fades — or is value gated behind setup, forms, config, tutorials, or a paywall?
- Is there a genuine "starter step" / tiny version (two-minute-rule) that leads naturally into the deeper behaviour — as opposed to a trivial action disconnected from real value?
- Are defaults, presets, saved state, and pre-filled data doing the work so recurring use is faster than first use? Or does every session reset the user to beginner mode?
- Does the product prioritise *doing the real behaviour early* over explaining/exploring? Is motion (setup, browsing) being mistaken for the action?
- Is the desired action easier than the competing behaviour (the user's current workaround, or the tempting distraction)?

## 4. Reward — is completing it immediately satisfying?  *(engagement, behaviour)*

- Does a reward arrive **quickly** after the action, close the loop, and actually satisfy the need that triggered it (not a decorative badge)?
- Is there **positive emotion at the moment of success** — a felt "that worked / I did it" — or an ambiguous ending where the user isn't sure they succeeded?
- Where it fits the product, is there **variability** tied to real value (rewards of the *tribe* — social validation; the *hunt* — information/resources/deals; the *self* — mastery/completion/control)? Does the reward type match the user's actual motivation?
- Does the reward reinforce the behaviour you want *repeated*, or does it reward a vanity action that creates no value? Does it leave a small open loop (curiosity about next time) without leaving the user empty or tricked?
- Is the product mistaking points/streaks/badges for genuine reward? (Extrinsic scaffolding is fine as support; it fails as the core engine — see the control-autonomy and ethics gates.)

## 5. Investment — does each cycle load the next?  *(adoption, engagement)*

The link the funnel-shaped mindset forgets. Asked *after* the user gets value, a small investment should make the next loop more likely and better:

- Does the user put something *in* — data, preferences, content, connections, reputation, history, configuration, progress — that makes the product better *for them* next time? Does that stored value make it harder to switch away (ethically, via real accumulated value, not hostage-taking)?
- Does investment **load the next trigger** (a scheduled item, a follow, a saved draft, a pending state that will prompt a return)?
- Does the product leave something motivating unfinished, and surface prior work at the return moment — or does it reset/waste what the user built?
- Is investment asked at the right time (after value, sized small) rather than demanded up front?

## 6. Tracking, progress & recovery  *(engagement, behaviour)*

- Is the behaviour that matters tracked and made **visible** (progress, history, milestones) so effort feels like it accumulates — without tracking vanity metrics that distort behaviour?
- Does progress feedback arrive soon enough to reinforce, and is it specific enough to guide the next action?
- Is there a **recovery path** after a lapse — "never miss twice," easy re-entry, restored context — rather than all-or-nothing streak mechanics that make one missed day cause abandonment?
- Does resuming after interruption drop the user back exactly where they were?

## 7. Identity & compounding  *(adoption)*

- Does repeated use cast a "vote" for an identity the user *wants* ("I'm someone who ships / trains / stays on top of X")? Does progress read as *becoming better*, not just task completion?
- Does the product avoid framing the user as broken/lazy/behind?
- Does the loop **strengthen with repetition** — easier and more valuable each cycle — and compound into value the user would miss if the product vanished? Or does it collapse once novelty fades or marketing pressure stops?

## Loop-failure diagnosis (use when behaviour isn't happening)

Walk the loop in order and find the *first* broken link — fixing a later link won't help if an earlier one is broken:

1. **No prompt** — nothing brings the user to the moment, or it's mistimed/too weak/ignored.
2. **Action too hard** — too many steps, too much thinking, value gated, needs skill/tools/confidence the user lacks, or asks too much too soon.
3. **Motivation too low** — the behaviour isn't wanted here, benefit is too abstract/distant, or a competing behaviour is easier. (Check this *last*.)
4. **Reward missing/delayed/unclear** — nothing satisfying follows, or the wrong action is rewarded.
5. **No investment** — value isn't stored, the next trigger isn't loaded, so the loop doesn't compound.

## Not your lens

Decision-point ergonomics and cognitive bias (framing, defaults, anchoring) → **cognitive-ease**. Whether the user gets genuinely *better* at something they care about → **capability-results**. Whether the loop respects user autonomy / crosses into manipulation → **control-autonomy** and the ethics gate. If you spot something there, note it in one line and move on — don't score it here.
